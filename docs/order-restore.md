# Восстановление отменённого заказа

## 1. Цель

Описать логику восстановления заказа из любого `cancelled_*` статуса обратно в активный статус (`new` / `processing` / `ready` / `in_delivery`).

Функция даёт модератору возможность откатить отмену, если ситуация изменилась (например, товар «нашёлся» на складе).

---

## 2. Сценарии использования

| Кейс | Откуда статус | Куда восстанавливаем |
|---|---|---|
| Отменили при оформлении — товар нашёлся | `cancelled_new` | `new` |
| Отменили при сборке — товар нашли | `cancelled_assembly` | `processing` |
| Отменили перед выдачей — передумали | `cancelled_ready` | `ready` |
| Отменили в доставке — возобновили | `cancelled_delivery` | `in_delivery` |
| Отменил клиент — клиент передумал | `cancelled_customer` | `new` |

---

## 3. Принцип работы

```
cancelled_*
    │
    ▼
Кнопка «Восстановить»
    │
    ├── Бэк: проверить остатки (stock) по каждому товару
    │
    ├── Если остатков достаточно:
    │       ├── Снять флаг отмены, вернуть активный статус
    │       ├── Пересоздать reservations (зарезервировать товары)
    │       ├── Очистить rejected_reason
    │       └── Вернуть заказ в работу
    │
    └── Если остатков не хватает:
            └── Вернуть ошибку: «Товар "X" доступен в количестве Y, требуется Z»
```

### 3.1. Проверка остатков

При восстановлении остатки могут измениться. Бэк должен:

1. По каждому `order_product` проверить `product_stock` на складе заказа
2. Если остатка хватает — зарезервировать (`quantity` + reserved)
3. Если не хватает — не резервировать, вернуть ошибку по каждому товару

Алгоритм аналогичен тому, что происходит при **создании заказа** — это переиспользование той же бизнес-логики.

### 3.2. Сброс `rejected_reason`

После восстановления причину отмены нужно очистить, чтобы она не смущала.

## 4. API

### Новый эндпоинт

```
POST /orders/restore/{id}
```

**Response (успех):**
```json
{
  "status": "ok",
  "data": {
    "id": 123,
    "order_status": "new",
    "rejected_reason": null
  }
}
```

**Response (ошибка остатков):**
```json
{
  "status": "error",
  "message": "Недостаточно товаров для восстановления заказа",
  "errors": [
    {
      "product_id": 456,
      "name": "Кроссовки Nike Air",
      "available": 2,
      "required": 3
    },
    {
      "product_id": 789,
      "name": "Футболка Puma",
      "available": 0,
      "required": 1
    }
  ]
}
```

## 5. Фронтенд

### 5.1. Server Action

Файл: `src/app/orders/edit/[id]/action.ts`

```typescript
export const restoreOrderAction = async (id: number) => {
  const cookieStore = await cookies();

  return await fetchService
    .post<null>({
      url: `orders/restore/${id}`,
    })
    .then((response) => {
      if (response.tokens) {
        updateTokensInAction(cookieStore, response.tokens);
      }

      revalidatePath(`/orders/edit/${id}`);
      revalidatePath("/orders");

      return response;
    });
};
```

### 5.2. Кнопка в `OrderStatusActions`

Показывать для всех `cancelled_*` статусов:

```typescript
const isCancelled = props.status.startsWith("cancelled_");

{isCancelled && (
  <Button onClick={handleRestore}>
    Восстановить заказ
  </Button>
)}
```

### 5.3. Обработка ошибок

Если бэк вернул 422 с ошибками по товарам — показать список проблемных товаров:

```
Товар "Кроссовки Nike Air" — доступно 2 шт, требуется 3
Товар "Футболка Puma" — доступно 0 шт, требуется 1
```

## 6. Маппинг статусов восстановления

| Текущий статус | Статус после восстановления | Примечание |
|---|---|---|
| `cancelled_new` | `new` | Заказ как новый, можно менять склад |
| `cancelled_assembly` | `processing` | Продолжить сборку |
| `cancelled_ready` | `ready` | Готов к выдаче/доставке |
| `cancelled_delivery` | `in_delivery` | Передать курьеру |
| `cancelled_customer` | `new` | Клиент передумал — заказ как новый |

---

## 7. Ограничения

- Нельзя восстановить заказ, если **все** товары недоступны — только частичная нехватка допустима (на усмотрение бизнес-логики)
- Если товар снят с продажи (удалён) — восстановление невозможно
- `completed` заказы не восстанавливаются (логически не имеет смысла)
- После восстановления заказ проходит стандартный флоу статусов
