# Переход статуса: `new` → `cancelled_new` / `processing`

## 1. Цель

Описать логику и архитектуру перехода заказа из статуса `new` в один из двух статусов:
- **`cancelled_new`** — отмена заказа админом с указанием причины
- **`processing`** — начало сборки заказа (с возможным предварительным созданием перемещений товаров между складами)

А также заложить архитектуру страницы создания перемещений, которая используется для двух типов:
- **Склад → Склад** — перемещение товаров между складами (на этапе `new → processing`)
- **Склад → Клиент** — отправка товаров курьером клиенту (на этапе `ready → in_delivery`)

---

## 2. Изменения в модели данных

### 2.1. OrderModel — добавлено поле `warehouse_id`

Главный склад определяется на этапе создания заказа (на бекенде):

- **Самовывоз** (`method_receipt: "pickup"`) — склад, который выбрал клиент
- **Доставка** (`method_receipt: "courier"`) — склад, ближайший по координатам к адресу доставки

```typescript
type OrderModel = {
  // ... существующие поля
  warehouse_id: number;       // ← новое: ID главного склада (выдача/доставка)
  warehouse?: WarehouseModel; // ← новое: объект склада (опционально, для отображения названия)
};
```

### 2.2. OrderProductModel.reservations — добавлено поле `warehouse_id`

Расширяем каждый элемент резерва, чтобы сразу знать, на каком складе сделан резерв (без дополнительного запроса к `product-stock`).

```typescript
type OrderProductModel = {
  // ... существующие поля
  reservations: {
    quantity: number;      // Количество, зарезервированное на данном остатке
    stock_id: number;      // ID остатка (product-stock)
    warehouse_id: number;  // ← новое: ID склада, которому принадлежит stock_id
  }[];
};
```

## 3. Определение главного склада

Значение `order.warehouse_id` приходит с бекенда в составе `OrderModel`. На фронте оно используется только для сравнения:

- Если `reservation.warehouse_id === order.warehouse_id` → товар уже на главном складе
- Если `reservation.warehouse_id !== order.warehouse_id` → товар нужно переместить на главный склад

## 4. Алгоритм анализа при загрузке страницы

### 4.1. Входные данные

Страница `orders/edit/[id]` (server action `fetchOrderEditPage`) получает:

```typescript
order: OrderModel & { warehouse_id: number };
products: (OrderProductModel & { reservations: { quantity: number; stock_id: number; warehouse_id: number }[] })[];
```

### 4.2. Вычисление необходимости перемещений

```typescript
type TransferNeedItem = {
  product_id: number;
  product_name: string;
  from_warehouse_id: number;
  from_warehouse_name: string;
  to_warehouse_id: number; // = order.warehouse_id
  to_warehouse_name: string;
  quantity: number;        // сколько нужно переместить
  stock_id: number;        // ID остатка-источника
};

function analyzeTransferNeeds(
  order: OrderModel,
  products: OrderProductModel[]
): {
  needsTransfer: boolean;
  transferItems: TransferNeedItem[];
} {
  const items: TransferNeedItem[] = [];

  for (const product of products) {
    for (const reservation of product.reservations) {
      if (reservation.warehouse_id !== order.warehouse_id) {
        items.push({
          product_id: product.product_id,
          product_name: product.name,
          from_warehouse_id: reservation.warehouse_id,
          from_warehouse_name: reservation.warehouse_name ?? `Склад #${reservation.warehouse_id}`,
          to_warehouse_id: order.warehouse_id,
          to_warehouse_name: order.warehouse?.name ?? `Склад #${order.warehouse_id}`,
          quantity: reservation.quantity,
          stock_id: reservation.stock_id,
        });
      }
    }
  }

  return {
    needsTransfer: items.length > 0,
    transferItems: items,
  };
}
```

## 5. UI на странице редактирования (`new`)

На странице `orders/edit/[id]/page.tsx` изменяется существующая таблица товаров и добавляется блок действий.

### 5.1. Колонка «Наличие на главном складе» в OrderProductsTable

В существующую таблицу `OrderProductsTable` добавляется колонка **«Наличие»**, где для каждого товара отображается статус его резервов относительно главного склада (`order.warehouse_id`):

| Статус | Отображение | Цвет |
|--------|------------|------|
| Все резервы товара на главном складе | `Есть на складе` | Зелёный (`#16a34a`) |
| Часть или все резервы на других складах | `Надо переместить N шт` | Красный (`#dc2626`) |

**Логика расчёта для одной строки товара:**

```typescript
function getAvailabilityStatus(
  product: OrderProductModel,
  mainWarehouseId: number
): { status: "available" | "need_transfer"; transferQty: number } {
  let transferQty = 0;

  for (const res of product.reservations) {
    if (res.warehouse_id !== mainWarehouseId) {
      transferQty += res.quantity;
    }
  }

  return {
    status: transferQty === 0 ? "available" : "need_transfer",
    transferQty,
  };
}
```

Пример отображения таблицы:

```
┌──────────┬────────────┬──────────┬──────────────────────────┐
│ Название │ Штрих-код  │ Кол-во   │ Наличие                  │
├──────────┼────────────┼──────────┼──────────────────────────┤
│ Товар А  │ 123456789  │ 10       │ Есть на складе           │ ✅ зелёный
│ Товар Б  │ 987654321  │ 5        │ Надо переместить 2 шт    │ 🔴 красный
└──────────┴────────────┴──────────┴──────────────────────────┘
```

### 5.2. Блок действий (OrderStatusActions)

Под таблицей товаров добавляется блок действий — клиентский компонент:

#### Сценарий A — Все товары `available`

```
[ 🟢 Начать сборку ]  [ 🔴 Отменить заказ ]
```

- **«Начать сборку»** → `PATCH /orders/{id}/status` → тело `{ status: "processing" }`
- **«Отменить заказ»** → модалка с полем `rejected_reason` → `cancelled_new`

#### Сценарий B — Есть товары `need_transfer`

```
⚠️ Некоторые товары забронированы на других складах

[ 📦 Создать перемещения ]  [ 🔴 Отменить заказ ]
```

- **«Создать перемещения»** → открывает модалку/страницу создания перемещений (см. раздел 6). Кнопка «Начать сборку» не показывается, пока не будут созданы все необходимые перемещения.
- **«Отменить заказ»** → модалка с полем `rejected_reason` → `cancelled_new`

> **Важно:** Таблица созданных перемещений будет отображаться на следующих статусах (например, `processing`), когда перемещения уже существуют. На этапе `new` показывается только индикация необходимости перемещений.

## 6. Страница создания перемещения

Создание перемещения — отдельная страница `/orders/transfer/create?order_id=123`. Она универсальна и поддерживает два типа перемещений:

| Тип | Откуда | Куда | Когда используется |
|-----|--------|------|-------------------|
| **Склад → Склад** | Склад-источник (где есть резерв) | Главный склад заказа (`order.warehouse_id`) | При переходе `new → processing`, если товары зарезервированы не на главном складе |
| **Склад → Клиент** | Главный склад заказа (`order.warehouse_id`) | Адрес клиента (`order.address`) | При переходе `ready → in_delivery`, когда заказ собран и нужна доставка курьером |

Страница получает параметр `type` (или определяет его автоматически по статусу заказа):
- `?order_id=123&type=stock_to_stock` — для перемещения между складами
- `?order_id=123&type=delivery` — для доставки курьером

### 6.1. Интерфейс страницы (тип «Склад → Склад»)

При переходе со статуса `new`, когда `analyzeTransferNeeds()` выявил товары на других складах:

```
Создание перемещения — Заказ #12345

Главный склад: Южно-Московский (все товары должны быть здесь)
Тип: Перемещение между складами

┌──────────────────────────────────────────────────────────────────┐
│ Перемещение #1                                                   │
│ Откуда: [ Склад Рязанский (id:12) ▼ ]  →  Куда: Южно-Московский │
│                                                                  │
│ Товары:                                                          │
│ ┌──────────┬─────────┬────────────────────────────────────────┐ │
│ │ Товар    │ Кол-во  │ Склад-источник                         │ │
│ ├──────────┼─────────┼────────────────────────────────────────┤ │
│ │ Товар А  │ 2 шт    │ Склад Рязанский (доступно: 10)         │ │
│ │ Товар Б  │ 5 шт    │ Склад Рязанский (доступно: 8)          │ │
│ └──────────┴─────────┴────────────────────────────────────────┘ │
│                                                                  │
│ [ + Добавить перемещение ]  [ 📦 Создать ]                      │
└──────────────────────────────────────────────────────────────────┘
```

**Ключевые моменты:**
- Склад «Откуда» автоматически подставляется из `reservation.warehouse_id`, но админ может изменить на другой, где `available >= quantity`
- Одно перемещение = одна пара «откуда → куда» с набором товаров
- Если товары распределены по 2 разным складам — создаётся 2 перемещения (админ видит оба и может редактировать)
- Кнопка «+ Добавить перемещение» — если нужно добавить ещё одну пару «откуда → куда»

### 6.2. Интерфейс страницы (тип «Склад → Клиент»)

При переходе со статуса `ready`, когда заказ собран и需要进行 доставка:

```
Создание отправления — Заказ #12345

Тип: Доставка курьером
Склад отправки: Южно-Московский
Адрес доставки: ул. Ленина, д. 10, кв. 5 (главный склад определён по lng/lat)

Карта: [маркер склада] → [маркер адреса клиента] ← Mapbox

┌──────────────────────────────────────┐
│ Товары к отправке:                    │
│ ┌──────────┬─────────┬──────────────┐ │
│ │ Товар    │ Кол-во  │ Цена         │ │
│ ├──────────┼─────────┼──────────────┤ │
│ │ Товар А  │ 10 шт   │ 5 000 ₽      │ │
│ │ Товар Б  │ 5 шт    │ 2 500 ₽      │ │
│ └──────────┴─────────┴──────────────┘ │
│                                       │
│ Ценность отправления: 7 500 ₽         │
│                                       │
│ [ Отправить ]                         │
└──────────────────────────────────────┘
```

**Ключевые моменты:**
- Склад отправки = `order.warehouse_id` (главный склад)
- Адрес доставки = `order.address` (отображается на карте)
- Карта Mapbox показывает маршрут от склада до адреса
- Кол-во товаров = общее количество в заказе (все товары уже собраны на главном складе)

### 6.3. Валидация перед созданием

**Для типа «Склад → Склад»:**

```typescript
type CreateTransferPayload = {
  order_id: number;
  type: "stock_to_stock";
  transfers: {
    from_warehouse_id: number;
    to_warehouse_id: number; // = order.warehouse_id
    items: {
      product_id: number;
      quantity: number;
      stock_id: number;
    }[];
  }[];
};
```

Проверки:
1. Все `TransferNeedItem` из `analyzeTransferNeeds()` назначены хотя бы в одно перемещение
2. На складе-источнике `available >= quantity` для каждого товара
3. `to_warehouse_id === order.warehouse_id` (все перемещения ведут на главный склад)

**Для типа «Склад → Клиент»:**

```typescript
type CreateDeliveryPayload = {
  order_id: number;
  type: "delivery";
  from_warehouse_id: number; // = order.warehouse_id
  to_address: AddressItem;   // = order.address
  items: {
    product_id: number;
    quantity: number;
  }[];
};
```

### 6.4. После создания перемещений

**После stock_to_stock:** после успешного создания ВСЕХ необходимых перемещений статус заказа автоматически меняется на `processing` (фронт отправляет `PATCH /orders/{id}/status`).

**После delivery:** после создания отправления статус меняется на `in_delivery`.

## 8. API Endpoints (предполагаемые)

| Метод | URL | Назначение |
|-------|-----|------------|
| `PATCH` | `/orders/{id}/status` | Смена статуса заказа. Body: `{ status: OrderStatus, rejected_reason?: string }` |
| `POST` | `/transfers/create` | Создание перемещения (stock_to_stock) |
| `POST` | `/deliveries/create` | Создание отправления курьером (delivery) |
| `GET` | `/warehouses/list-by-ids?ids=1,2,3` | Получить данные складов для отображения названий |

---

## 9. Примечания

- Резервы (`reservations`) создаются на этапе оформления заказа (бекенд). Фронт их только читает.
- При переходе в `processing` никакие остатки повторно не проверяются — резервы уже гарантируют наличие.
- Если админ отменяет заказ (`cancelled_new`), резервы должны быть освобождены (бекенд).
- При создании перемещений система должна «перевешивать» резерв с одного `stock_id` на другой (после прибытия товара на главный склад) — ответственность бекенда.
- Таблица созданных перемещений отображается на этапе `processing` (и далее), когда перемещения уже существуют. На этапе `new` показывается только колонка-индикатор в `OrderProductsTable`.
- Страница `/orders/transfer/create` — универсальная: определяет тип перемещения по параметру `type` или по статусу заказа.
