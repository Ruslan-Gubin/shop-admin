# Логика выбора цены

## Определение типа клиента

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | ID клиента |
| name | string | Имя клиента |
| type | enum | retail / wholesale / employee |

## Алгоритм выбора цены

### Розничный клиент

```
1. Определяем базовую цену по количеству:
   ЕСЛИ quantity ≥ 100 И есть "Крупный опт"
       → basePrice = "Крупный опт"
   ИНАЧЕ ЕСЛИ quantity ≥ 10 И есть "Мелкий опт"
       → basePrice = "Мелкий опт"
   ИНАЧЕ
       → basePrice = "Розничная"

2. Проверяем isActive для базового типа

3. Результат:
   → клиент видит цену базового типа
```

### Пример (розничный)

```
Товар: iPhone 15
- Розничная: 89 000 ₽
- Мелкий опт (от 10 шт): 82 000 ₽
- Крупный опт (от 100 шт): 78 000 ₽

Клиент: розничный, quantity = 5 шт
→ basePrice = "Розничная" = 89 000 ₽
```

### Пример 2 (оптовая цена выгоднее)

```
Клиент: розничный, quantity = 15 шт
→ basePrice = "Мелкий опт" = 82 000 ₽
```

### Оптовый клиент

```
ЕСЛИ клиент.type = "wholesale"
    → показать "Оптовая продажа" (всегда, игнорировать количество)
```

### Клиент-сотрудник

```
ЕСЛИ клиент.type = "employee"
    → показать "Сотрудник"
```

## Приоритеты

| Приоритет | Тип клиента | Логика |
|-----------|-------------|--------|
| 1 | Оптовый | Всегда своя цена |
| 2 | Сотрудник | Всегда своя цена |
| 3 | Розничный | Цена зависит от количества |

## Fallback логика

Когда цена не заполнена для товара, работает цепочка fallback:

```
Нет "Мелкий опт" → "Розничная"
Нет "Крупный опт" → "Мелкий опт" → иначе "Розница"
Нет "Оптовая продажа" → fallback на самый дешевый опт
```

### Правило Fallback

При отсутствии цены для типа X берётся:
1. Ближайший тип с меньшим minQuantity
2. Если нет — ближайший с большим minQuantity
3. Если нет вообще — требовать заполнение
